# 04. Правила работы с Astro

Этот файл описывает, как Codex должен работать с Astro в проектах лендингов, корпоративных сайтов и контентных страниц.

## Когда использовать Astro

Astro нужно использовать как основной инструмент для сайтов, где важны:

- быстрый первый экран;
- SEO;
- статическая генерация;
- удобная структура страниц;
- минимум клиентского JavaScript;
- простая интеграция с React-компонентами;
- удобная работа с контентом и шаблонами.

Для лендингов и корпоративных сайтов Astro обычно должен быть первым выбором, если не требуется полноценное SPA.

## Главный принцип

Astro-проект должен отдавать максимум полезного HTML без лишней гидрации.

Codex должен соблюдать:

- KISS: простая структура лучше лишней архитектуры;
- YAGNI: не добавлять сложные слои заранее;
- правила дизайн-системы из `instructions/01-design-system.md`;
- правила React из `instructions/02-react-rules.md`, если используются React-острова;
- общие правила из `instructions/03-general-programming.md`.

Не нужно превращать Astro-сайт в React-приложение без причины.

## Структура проекта

Если проект уже имеет структуру, следуй ей.

Типовая структура:

```text
src/pages
src/layouts
src/components
src/components/react
src/styles
src/content
src/data
src/lib
public
```

Назначение папок:

- `src/pages` — маршруты сайта;
- `src/layouts` — базовые шаблоны страниц;
- `src/components` — Astro-компоненты;
- `src/components/react` — React-компоненты для интерактивных островов;
- `src/styles` — глобальные стили, токены, базовая типографика;
- `src/content` — контентные коллекции Astro;
- `src/data` — простые данные для секций, меню, тарифов, FAQ;
- `src/lib` — вспомогательные функции и интеграции;
- `public` — статические файлы, которые должны отдаваться как есть.

Не создавай новую структуру, если уже есть понятное место для файла.

## Страницы

Страницы в `src/pages` должны быть тонкими.

Хорошо, когда страница:

- собирает секции;
- передает данные в компоненты;
- задает SEO-метаданные;
- подключает layout;
- не содержит большой бизнес-логики.

Если страница становится слишком длинной, вынеси повторяемые части в компоненты или данные.

## Layouts

Layout должен задавать общий каркас страницы:

- `<html lang="">`;
- `<head>`;
- базовые meta-теги;
- header;
- footer;
- подключение глобальных стилей;
- общую разметку страницы;
- SEO-данные, которые приходят через props.

Не нужно дублировать `<head>`, header и footer на каждой странице.

## Компоненты Astro

Astro-компоненты должны быть основой интерфейса.

Используй `.astro` для:

- секций лендинга;
- карточек;
- навигации;
- футера;
- hero-блоков;
- списков услуг;
- FAQ;
- хлебных крошек;
- статических форм без сложной клиентской логики.

React подключай только для реальной интерактивности.

## React-острова

React-компоненты в Astro нужно гидрировать только там, где это нужно.

Правила:

- `client:load` — только если компонент нужен сразу после загрузки страницы;
- `client:idle` — для второстепенных интерактивных элементов;
- `client:visible` — для компонентов ниже первого экрана;
- `client:media` — если компонент нужен только на определенном размере экрана;
- `client:only` — только если компонент не может рендериться на сервере.

Пример:

```astro
---
import PriceCalculator from "../components/react/PriceCalculator";
---

<PriceCalculator client:visible />
```

Не нужно ставить `client:load` по привычке.

## Данные

Повторяемый контент лучше выносить в данные.

Подходит для:

- меню;
- услуг;
- тарифов;
- преимуществ;
- этапов работы;
- отзывов;
- FAQ;
- контактов;
- регионов;
- карточек кейсов.

Пример:

```ts
export const services = [
  {
    title: "Разработка сайта",
    description: "Лендинг или корпоративный сайт под задачи бизнеса.",
  },
];
```

Не нужно выносить все подряд в данные, если контент используется один раз и так читается лучше.

## Content Collections

Astro Content Collections стоит использовать, если в проекте есть много однотипного контента:

- статьи;
- новости;
- кейсы;
- услуги;
- города;
- специалисты;
- вакансии;
- документация.

Для коллекций нужно описывать schema, чтобы данные были проверяемыми.

Пример:

```ts
import { defineCollection, z } from "astro:content";

const cases = defineCollection({
  schema: z.object({
    title: z.string(),
    description: z.string(),
    date: z.date(),
  }),
});

export const collections = { cases };
```

Не используй Content Collections для маленького лендинга, если обычного массива данных достаточно.

## Стили

Стили Astro-проекта должны соответствовать дизайн-системе.

Базовые файлы:

```text
src/styles/tokens.css
src/styles/global.css
```

Правила:

- дизайн-токены хранить в CSS-переменных;
- глобальные стили держать в одном понятном месте;
- стили компонента писать внутри `.astro`, если они относятся только к нему;
- повторяемые классы выносить в общий CSS;
- не дублировать одинаковые отступы и цвета вручную;
- не добавлять новый визуальный стиль без обновления дизайн-системы.

Scoped-стили Astro удобны для секций и небольших компонентов. Но если стиль должен переиспользоваться, лучше вынести его в общий слой.

## Изображения

Изображения должны быть оптимизированы.

Правила:

- использовать `astro:assets`, если изображение лежит в `src`;
- хранить публичные статические файлы в `public`;
- у важных изображений должен быть понятный `alt`;
- декоративным изображениям можно задавать пустой `alt`;
- не загружать изображения большего размера, чем нужно;
- использовать lazy loading для изображений вне первого экрана;
- задавать размеры или aspect-ratio, чтобы страница не прыгала при загрузке.

Для первого экрана важное изображение должно быть достаточно легким и не ломать LCP.

## SEO

Astro должен отдавать важный SEO-контент в HTML.

На каждой странице должны быть:

- уникальный `title`;
- уникальный `description`;
- один основной `h1`;
- понятная структура `h2` и `h3`;
- canonical URL, если это нужно проекту;
- Open Graph-данные для важных страниц;
- корректный `lang`;
- человекочитаемые URL;
- внутренние ссылки между важными страницами.

Не нужно генерировать важный SEO-текст только на клиенте.

Подробные правила SEO и GEO описываются в отдельной инструкции.

## Маршруты

Маршруты должны быть короткими, понятными и стабильными.

Хорошо:

```text
/services/
/services/site-development/
/cases/
/contacts/
/about/
```

Плохо:

```text
/page1/
/new-section-final/
/services2/
```

Если меняется URL существующей страницы, нужно подумать о редиректе.

## Sitemap и robots

Для продакшен-сайта нужно предусмотреть:

- `sitemap.xml`;
- `robots.txt`;
- правильный `site` в `astro.config.*`;
- запрет индексации тестовых окружений;
- отсутствие служебных страниц в sitemap.

Если сайт мультирегиональный или многостраничный, sitemap должен учитывать все важные страницы.

## Производительность

Astro-проект должен оставаться легким.

Правила:

- не гидрировать компоненты без необходимости;
- не добавлять тяжелые библиотеки ради простых эффектов;
- не подключать глобальный JavaScript для локальной задачи;
- не грузить все изображения сразу;
- не использовать видео как фон без оптимизации;
- не подключать несколько шрифтов и начертаний без причины;
- проверять, что первый экран не перегружен.

Для лендингов и корпоративных сайтов особенно важны:

- LCP;
- CLS;
- INP;
- вес JavaScript;
- размер изображений;
- скорость ответа сервера.

## Формы

Формы в Astro могут быть:

- обычными HTML-формами;
- Astro endpoint;
- React-компонентом, если нужна сложная интерактивность;
- отправкой на PHP-обработчик на сервере.

Обязательные правила:

- валидировать данные на клиенте для удобства;
- валидировать данные на сервере для безопасности;
- не отправлять лишние поля;
- не хранить персональные данные в клиентском хранилище;
- показывать понятный результат отправки;
- защищать формы от спама;
- не раскрывать адреса webhook и секретные ключи на клиенте.

Если форма отправляется на PHP, клиентский код должен обращаться только к публичному endpoint, а секреты должны оставаться на сервере.

## Переменные окружения

Секреты нельзя использовать в клиентском коде.

Правила:

- приватные переменные использовать только на сервере;
- публичные переменные помечать по правилам Astro;
- не вставлять токены и webhook URL в компоненты;
- хранить пример переменных в `.env.example` без реальных значений;
- проверять, что тестовые ключи не попали в репозиторий.

Примеры:

```text
PUBLIC_SITE_URL=https://example.com
CRM_WEBHOOK_URL=
MAIL_HOST=
MAIL_USER=
MAIL_PASS=
```

`PUBLIC_*` можно считать видимым в браузере, поэтому туда нельзя класть секреты.

## Интеграции

Перед добавлением интеграции проверь, нужна ли она реально.

Частые интеграции:

- `@astrojs/react`;
- `@astrojs/sitemap`;
- `@astrojs/mdx`;
- адаптер для SSR, если сайт не полностью статический.

Не нужно добавлять SSR-адаптер, если сайт может быть статическим.

## SSR и статическая сборка

По умолчанию для лендингов и корпоративных сайтов лучше выбирать статическую сборку.

SSR нужен, если есть:

- авторизация;
- личный кабинет;
- серверная персонализация;
- закрытые данные;
- динамические ответы API;
- серверная логика, которую нельзя выполнить на этапе сборки.

Если SSR не нужен, не включай его просто на всякий случай.

## Проверка перед завершением задачи

После изменений в Astro-проекте нужно запустить доступные проверки:

```bash
npm run lint
npm run typecheck
npm run build
```

Если скриптов нет, нужно хотя бы запустить сборку или явно написать, что проверки недоступны.

Для UI-задач дополнительно проверить:

- desktop;
- mobile;
- отсутствие наложений текста;
- корректные изображения;
- доступность интерактивных элементов;
- что важный контент есть в HTML;
- что нет лишней гидрации.

## Финальная проверка Astro-задачи

Перед завершением задачи Codex должен убедиться:

- Astro используется как статическая основа сайта;
- React подключен только для нужной интерактивности;
- важный контент доступен без клиентского JavaScript;
- SEO-метаданные заданы на уровне страницы или layout;
- изображения оптимизированы;
- стили соответствуют дизайн-системе;
- секреты не попали в клиентский код;
- сборка и доступные проверки выполнены или причина их отсутствия указана в ответе.
