# 02. Правила работы на React

Этот файл описывает, как Codex должен писать React-код в проектах с лендингами, корпоративными сайтами и интерактивными блоками.

## Когда использовать React

React нужно использовать там, где есть настоящая интерактивность:

- сложные формы;
- калькуляторы;
- фильтры и поиск;
- квизы;
- личные кабинеты;
- интерактивные карты;
- слайдеры со сложной логикой;
- виджеты, которые активно меняют состояние;
- компоненты, которые требуют клиентского состояния.

Не нужно использовать React для простой статической верстки, если достаточно HTML, Astro-компонента и CSS.

Для Astro-проектов React должен быть островом интерактивности, а не причиной превращать весь сайт в SPA.

## Главный принцип

React-код должен быть простым, предсказуемым и локальным.

Codex должен соблюдать:

- KISS: не усложнять компоненты без необходимости;
- YAGNI: не добавлять функциональность "на будущее";
- единый стиль проекта;
- правила из `instructions/01-design-system.md`;
- общие правила из `instructions/03-general-programming.md`.

Если React нужен только для небольшой интерактивности, компонент должен оставаться небольшим и понятным.

## Структура компонентов

Компоненты нужно размещать рядом с существующими React-компонентами проекта.

Обычно используются такие папки:

```text
src/components
src/components/react
src/widgets
src/features
src/ui
```

Если в проекте уже есть другая структура, следуй ей.

Один компонент не должен становиться файлом на сотни строк. Если компонент разрастается, можно вынести:

- маленькие дочерние компоненты;
- чистые функции;
- константы;
- типы;
- хуки;
- работу с API.

Но не нужно дробить код заранее, если компонент остается простым.

## Имена

Правила именования:

- React-компоненты называются в `PascalCase`;
- хуки начинаются с `use`;
- обработчики событий называются от действия: `handleSubmit`, `handleChange`, `handleClose`;
- boolean-переменные начинаются с `is`, `has`, `can` или `should`;
- файлы компонентов называются понятно и совпадают по смыслу с компонентом.

Пример:

```tsx
ContactForm.tsx
PriceCalculator.tsx
CaseFilter.tsx
useLeadForm.ts
```

## Props

Props должны быть явными и понятными.

Хорошо:

```tsx
type ButtonProps = {
  children: React.ReactNode;
  variant?: "primary" | "secondary";
  disabled?: boolean;
  onClick?: () => void;
};
```

Плохо:

```tsx
type ButtonProps = {
  data: any;
  config: any;
};
```

Не передавай в компонент огромный объект, если компоненту нужны только 2-3 поля.

## TypeScript

Если проект использует TypeScript, React-код тоже должен быть типизирован.

Правила:

- не использовать `any`, если можно описать тип;
- типизировать props;
- типизировать данные форм;
- типизировать ответы API;
- типизировать настройки компонентов;
- не усложнять типы без необходимости.

Для простых компонентов достаточно локального `type Props`.

Общие типы можно выносить в отдельный файл, если они используются в нескольких местах.

## Состояние

Состояние должно жить как можно ближе к месту использования.

Используй `useState`, когда состояние простое:

- открыто/закрыто;
- выбранный пункт;
- значение поля;
- статус отправки формы;
- текущий шаг.

Используй `useReducer`, когда:

- состояние состоит из нескольких связанных полей;
- есть несколько типов действий;
- обновления сложно читать через много `setState`.

Не добавляй глобальное состояние, если данные нужны только одному компоненту или одной секции.

## Эффекты

`useEffect` нужно использовать только для синхронизации с внешним миром:

- запросы к API;
- подписки;
- таймеры;
- работа с `localStorage`;
- работа с `window`, `document`, `IntersectionObserver`;
- подключение сторонних виджетов.

Не используй `useEffect` для вычислений, которые можно сделать во время рендера.

Каждый эффект должен иметь понятные зависимости и cleanup, если он создает подписку, таймер или слушатель события.

## Производительность

Для лендингов и корпоративных сайтов React не должен утяжелять страницу без причины.

Правила:

- не подключать React-компонент там, где достаточно статического HTML;
- не хранить в состоянии данные, которые можно вычислить из props;
- не делать повторные API-запросы при каждом рендере;
- не добавлять тяжелые библиотеки ради маленького виджета;
- использовать lazy loading для тяжелых клиентских компонентов;
- мемоизацию применять только там, где есть реальная причина.

`useMemo` и `useCallback` не нужно добавлять автоматически. Они нужны, если есть измеримая проблема, тяжелое вычисление или стабильная ссылка действительно важна.

## Формы

Формы на React должны быть надежными.

Обязательные правила:

- валидировать обязательные поля;
- показывать понятные ошибки рядом с полями;
- блокировать повторную отправку;
- показывать состояние загрузки;
- показывать успешную отправку;
- не очищать форму при ошибке;
- не отправлять лишние поля;
- не хранить персональные данные в `localStorage`;
- дублировать критичную валидацию на сервере.

Для заявок с сайта нужно предусмотреть антиспам: honeypot, rate limit на сервере, captcha или другой способ, подходящий проекту.

## Работа с API

Если запрос используется больше одного раза, вынеси его из компонента.

Пример структуры:

```text
src/lib/api.ts
src/lib/forms.ts
src/services/leads.ts
```

Правила:

- обрабатывать ошибки сети;
- обрабатывать неуспешные HTTP-статусы;
- не показывать пользователю технический текст ошибки;
- не логировать токены, cookies, пароли и персональные данные;
- не хранить секреты в React-коде;
- брать публичные настройки только из разрешенных public env-переменных.

## Стили

Стили React-компонентов должны соответствовать дизайн-системе.

Предпочтительный порядок:

1. Использовать существующие компоненты и классы.
2. Использовать CSS-переменные проекта.
3. Добавить локальные стили компонента.
4. Создать новый общий компонент, если паттерн повторяется.

Не нужно добавлять новый визуальный стиль внутри React-компонента, если он не описан в дизайн-системе.

Компонент должен иметь состояния:

- `hover`, если элемент интерактивный;
- `focus`, если элемент доступен с клавиатуры;
- `disabled`, если действие может быть недоступно;
- `loading`, если есть ожидание запроса.

## Доступность

React-компоненты должны быть доступны.

Правила:

- использовать `button` для действий;
- использовать `a` для переходов;
- связывать `label` и поле формы;
- добавлять `aria-*` только там, где семантики HTML недостаточно;
- делать видимый focus;
- поддерживать управление с клавиатуры;
- закрывать модальные окна по `Escape`;
- возвращать фокус после закрытия модального окна;
- не блокировать масштабирование страницы;
- не делать важный текст только картинкой.

Если компонент меняет состояние асинхронно, пользователь должен понимать, что происходит.

## React в Astro

В Astro-проектах React-компоненты нужно подключать только с нужной директивой гидрации.

Правила:

- `client:load` использовать только для компонентов, нужных сразу;
- `client:idle` использовать для второстепенных виджетов;
- `client:visible` использовать для блоков ниже первого экрана;
- `client:only` использовать только если компонент не может рендериться на сервере;
- не гидрировать всю страницу без причины.

Пример:

```astro
---
import ContactForm from "../components/react/ContactForm";
---

<ContactForm client:visible />
```

Если интерактивность не нужна, компонент должен остаться `.astro`.

## SEO

React не должен мешать индексации сайта.

Для лендингов и корпоративных сайтов важный контент должен быть доступен в HTML:

- заголовки;
- основной текст;
- услуги;
- цены, если они открытые;
- контакты;
- адреса;
- FAQ;
- хлебные крошки;
- внутренние ссылки.

Не нужно прятать важный SEO-контент за клиентской загрузкой, если его можно отдать статически через Astro или сервер.

Подробные правила SEO и GEO описываются в отдельной инструкции.

## Безопасность

React-код не должен раскрывать секреты.

Запрещено:

- хранить API-ключи в клиентском коде;
- вставлять приватные webhook URL в React;
- выводить технические ошибки пользователю;
- логировать персональные данные;
- вставлять сырой HTML без проверки;
- доверять query-параметрам и данным из CMS.

`dangerouslySetInnerHTML` можно использовать только если есть явная причина и контент очищен безопасным способом.

## Тестирование и проверка

После изменения React-кода нужно запустить доступные проверки:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

Если в проекте нет таких скриптов, нужно проверить хотя бы сборку или явно написать, что проверки недоступны.

Для UI-задач нужно проверить:

- desktop;
- mobile;
- состояние загрузки;
- состояние ошибки;
- пустые данные;
- клавиатурную навигацию;
- отсутствие наложений текста и элементов.

## Финальная проверка React-задачи

Перед завершением задачи Codex должен убедиться:

- React используется только там, где он действительно нужен;
- компонент соответствует дизайн-системе;
- props и состояние понятны;
- нет лишних абстракций;
- нет секретов в клиентском коде;
- формы обрабатывают ошибки;
- важный SEO-контент не спрятан за клиентским рендером;
- доступные проверки выполнены или причина их отсутствия указана в ответе.
