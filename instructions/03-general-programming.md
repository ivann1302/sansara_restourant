# 03. Общие правила программирования

Этот файл описывает базовые правила разработки для всех проектов: React, Astro, PHP и любых вспомогательных скриптов.

## Главный принцип

Codex должен сначала понять проект, а потом менять код.

Перед изменениями нужно посмотреть:

- структуру проекта;
- `package.json`, `composer.json` или другие файлы зависимостей;
- существующие компоненты, стили и хелперы;
- правила форматирования;
- уже принятые архитектурные решения;
- README или документацию, если они есть.

Не нужно переписывать проект под другой стиль, если текущий стиль уже работает.

## Минимальные изменения

Любая правка должна быть настолько маленькой, насколько это возможно для решения задачи.

Codex не должен без явной необходимости:

- делать масштабный рефакторинг;
- менять архитектуру;
- переименовывать файлы;
- менять публичные API;
- обновлять зависимости;
- удалять код, который не относится к задаче;
- форматировать весь проект ради одной правки.

Если задача требует более крупного изменения, сначала нужно объяснить причину.

## KISS, YAGNI и SOLID

Codex должен соблюдать KISS и YAGNI как базовые принципы для всех задач.

KISS означает: решение должно быть простым, прямым и понятным. Не нужно усложнять код абстракциями, паттернами или универсальными механизмами, если задача решается простым способом.

YAGNI означает: не нужно добавлять функциональность, которая пока не требуется. Не создавай запасные настройки, расширения, слои, интерфейсы и универсальные решения только на случай будущих задач.

Правила применения:

- сначала выбирай самое простое решение, которое корректно закрывает задачу;
- добавляй абстракцию только тогда, когда она уменьшает реальную сложность или устраняет реальное повторение;
- не проектируй систему под гипотетические будущие требования;
- не дроби код на слишком много файлов без необходимости;
- не добавляй фабрики, классы, интерфейсы и конфиги, если обычная функция или компонент понятнее;
- не усложняй лендинг или корпоративный сайт архитектурой уровня большого продукта.

SOLID нужно применять там, где он действительно помогает:

- в серверном PHP-коде с бизнес-логикой;
- в сложных сервисах, интеграциях и обработчиках данных;
- в повторно используемых модулях;
- в коде, который будет расширяться и тестироваться отдельно.

SOLID не должен превращаться в самоцель. Если соблюдение принципа делает маленькую задачу сложнее, менее читаемой или добавляет лишние слои, нужно выбрать более простое решение.

## Читаемость кода

Код должен быть понятен человеку, который откроет проект через несколько месяцев.

Правила:

- имена переменных, функций и компонентов должны описывать смысл;
- функции должны делать одну понятную вещь;
- сложную логику лучше разбивать на небольшие части;
- одинаковую логику не нужно копировать в разные места;
- магические числа и строки нужно выносить в константы, если они имеют смысл в домене;
- комментарии нужны только там, где код сам по себе не объясняет причину решения.

Не нужно писать комментарии, которые просто пересказывают строку кода.

## Предсказуемая структура

Новые файлы нужно размещать рядом с похожими существующими файлами.

Если в проекте уже есть структура, следуй ей. Например:

```text
src/components
src/layouts
src/pages
src/styles
src/utils
src/lib
```

Не создавай новую папку, если в проекте уже есть подходящее место.

## Работа с зависимостями

Перед добавлением новой зависимости проверь, нельзя ли решить задачу средствами проекта или стандартной библиотеки.

Новую зависимость можно добавлять, если:

- она решает реальную задачу;
- она популярна и поддерживается;
- она не дублирует уже установленную библиотеку;
- ее размер и сложность оправданы;
- без нее код станет заметно хуже.

Для маленьких задач не нужно тащить большую библиотеку.

## Ошибки и крайние случаи

Код должен учитывать не только счастливый путь.

Проверяй:

- пустые данные;
- отсутствующие поля;
- ошибки сети;
- ошибки API;
- неверный формат данных;
- повторную отправку форм;
- медленную загрузку;
- отсутствие JavaScript там, где это важно для сайта;
- мобильные размеры экрана.

Ошибки должны обрабатываться явно, без скрытых падений.

## Типы и контракты данных

Если проект использует TypeScript, нужно описывать типы для данных, которые приходят извне:

- API-ответы;
- данные CMS;
- данные форм;
- настройки сайта;
- SEO-метаданные;
- конфиги компонентов.

Нельзя использовать `any`, если тип можно описать нормально.

Если проект на JavaScript без TypeScript, используй понятные имена, проверки данных и JSDoc там, где это помогает.

## Формы

Формы должны быть надежными и удобными.

Обязательные правила:

- валидировать данные на клиенте для удобства;
- валидировать данные на сервере для безопасности;
- показывать понятные ошибки;
- блокировать повторную отправку, пока запрос выполняется;
- не терять введенные данные при ошибке;
- не отправлять лишние поля;
- не хранить чувствительные данные в localStorage.

Если форма отправляет заявку, письмо или лид, нужно предусмотреть защиту от спама.

## Работа с API

Запросы к API должны быть вынесены в отдельный слой, если они используются больше одного раза.

Правила:

- не разбрасывать одинаковые `fetch`-запросы по компонентам;
- обрабатывать ошибки ответа и ошибки сети;
- не доверять данным, пришедшим снаружи;
- не показывать пользователю технические сообщения об ошибках;
- не логировать токены, пароли, cookies и персональные данные.

## Конфигурация и окружение

Все настройки, которые отличаются между окружениями, должны лежать в переменных окружения.

Примеры:

```text
API_URL
PUBLIC_SITE_URL
MAIL_HOST
MAIL_USER
MAIL_PASS
CRM_WEBHOOK_URL
```

Секретные значения нельзя хранить в репозитории.

Для примера можно создавать `.env.example`, но без реальных ключей и паролей.

## Git и чужие изменения

Codex не должен откатывать изменения, которые не относятся к текущей задаче.

Перед изменениями желательно проверить состояние репозитория:

```bash
git status --short
```

Если в файле уже есть чужие изменения, нужно работать аккуратно и не затирать их.

Запрещено без явной команды пользователя:

- делать `git reset --hard`;
- удалять чужие изменения;
- переписывать историю;
- чистить untracked-файлы;
- делать массовую автоформатировку всего проекта.

## Проверка результата

После изменений нужно запустить доступные проверки проекта.

В зависимости от проекта это может быть:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
composer test
php -l file.php
```

Если проверка не запускается или в проекте нет скриптов, нужно сказать об этом в ответе.

Если проверка падает из-за уже существующей ошибки, нужно отделить ее от своей правки.

## Производительность

Код не должен без причины ухудшать скорость сайта.

Правила:

- не добавлять тяжелые скрипты без необходимости;
- не загружать изображения большего размера, чем нужно;
- использовать lazy loading для медиа вне первого экрана;
- не выполнять дорогие вычисления при каждом рендере;
- не дублировать запросы к API;
- не подключать клиентский JavaScript там, где достаточно статического HTML.

Для лендингов и корпоративных сайтов особенно важны быстрый первый экран, легкие изображения и минимальный JavaScript.

## Доступность

Интерфейс должен быть доступен для клавиатуры и экранных читалок.

Минимальные правила:

- использовать семантические HTML-теги;
- у интерактивных элементов должны быть понятные названия;
- кнопки должны быть кнопками, ссылки должны быть ссылками;
- у изображений должен быть корректный `alt`;
- focus-состояние должно быть видимым;
- модальные окна должны управляться с клавиатуры;
- цвет не должен быть единственным способом передать смысл.

## Безопасность по умолчанию

Нельзя доверять данным от пользователя, CMS, API, query-параметров и cookies.

Минимальные правила:

- экранировать пользовательский контент;
- валидировать данные на сервере;
- не вставлять сырой HTML без необходимости;
- не хранить секреты в клиентском коде;
- не выводить stack trace пользователю;
- не логировать чувствительные данные;
- проверять загрузку файлов по типу, размеру и назначению.

Подробные правила безопасности лежат в отдельном файле инструкции по безопасности.

## Финальный ответ после задачи

После выполнения задачи Codex должен кратко написать:

- что изменено;
- в каких файлах;
- какие проверки запускались;
- есть ли ограничения или что осталось сделать.

Не нужно писать длинный отчет, если задача была маленькой.
