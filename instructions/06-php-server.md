# 06. Правила работы с PHP на сервере

Этот файл описывает, как Codex должен писать и поддерживать PHP-код на сервере для лендингов, корпоративных сайтов и небольших backend-задач.

## Когда использовать PHP

PHP на сервере нужен, если проекту требуется:

- обработка форм;
- отправка писем;
- отправка заявок в CRM;
- работа с webhook;
- простые API endpoint;
- server-side валидация;
- загрузка файлов;
- интеграция с платежами или внешними сервисами;
- закрытая серверная логика, которую нельзя отдавать в браузер.

Не нужно добавлять PHP, если задача решается статической сборкой Astro и не требует серверной логики.

## Главный принцип

PHP-код должен быть простым, безопасным и предсказуемым.

Codex должен соблюдать:

- KISS: не строить сложную архитектуру для одного обработчика формы;
- YAGNI: не добавлять слои и классы на будущее;
- SOLID там, где есть реальная бизнес-логика, сервисы и интеграции;
- общие правила из `instructions/03-general-programming.md`;
- правила безопасности из отдельной инструкции по безопасности.

Для маленького сайта допустим простой PHP-обработчик. Для нескольких форм, CRM, почты и логирования лучше вынести повторяемую логику в небольшие функции или сервисы.

## Структура PHP-файлов

Если в проекте уже есть структура, следуй ей.

Возможная структура для простого сайта:

```text
server
server/public
server/src
server/src/Handlers
server/src/Services
server/src/Support
server/config
server/logs
```

Назначение:

- `server/public` — публичные endpoint, доступные из браузера;
- `server/src` — внутренний PHP-код;
- `server/src/Handlers` — обработчики запросов;
- `server/src/Services` — отправка почты, CRM, интеграции;
- `server/src/Support` — валидация, ответы, утилиты;
- `server/config` — конфиги без секретов;
- `server/logs` — логи, если они нужны и закрыты от публичного доступа.

Если хостинг требует класть PHP прямо в публичную директорию, нужно особенно внимательно следить, чтобы рядом не лежали секреты, `.env`, логи и внутренние файлы.

## Публичная и приватная часть

В публичной директории должны лежать только файлы, которые действительно должны быть доступны из браузера.

Нельзя держать в публичной директории:

- `.env`;
- логи;
- резервные копии;
- дампы базы;
- конфиги с секретами;
- `vendor`, если сервер позволяет обращаться к нему напрямую;
- временные файлы загрузок;
- исходники с приватной логикой, если их можно вынести выше web root.

Если возможно, точкой входа должен быть один публичный файл, а внутренняя логика должна подключаться из приватной директории.

## Версия PHP

Перед написанием кода нужно понять, какая версия PHP доступна на сервере.

Правила:

- не использовать синтаксис новее версии PHP на хостинге;
- если версия неизвестна, писать совместимо с распространенной современной версией PHP 8.x;
- использовать строгую типизацию, если проект это поддерживает;
- не полагаться на расширения PHP, которых может не быть на сервере;
- проверять код через `php -l`, если PHP доступен локально.

В новых файлах желательно использовать:

```php
<?php

declare(strict_types=1);
```

## Composer

Composer можно использовать, если проект и хостинг это поддерживают.

Composer уместен для:

- автозагрузки классов;
- надежной отправки почты;
- работы с SDK внешних сервисов;
- валидации;
- тестов;
- dotenv в приватной серверной части.

Не нужно добавлять Composer и зависимости ради одного маленького endpoint, если хостинг и задача лучше решаются простым PHP.

Если Composer используется:

- `vendor` не должен быть доступен как публичная директория;
- `composer.lock` должен попадать в репозиторий;
- зависимости должны быть минимальными и поддерживаемыми;
- после изменения зависимостей нужно проверять проект на серверной версии PHP.

## Переменные окружения

Секреты должны храниться вне клиентского кода и вне публичной директории.

Примеры секретов:

```text
MAIL_HOST
MAIL_USER
MAIL_PASS
MAIL_FROM
CRM_WEBHOOK_URL
CRM_TOKEN
TELEGRAM_BOT_TOKEN
TELEGRAM_CHAT_ID
```

Правила:

- не коммитить реальные `.env`;
- держать `.env.example` без реальных значений;
- не выводить значения env в браузер;
- не логировать секреты;
- не передавать секреты в JavaScript;
- не хранить webhook URL в React или Astro-клиенте.

Если хостинг не поддерживает env напрямую, можно использовать приватный PHP-конфиг вне web root, но не хранить его в репозитории с реальными данными.

## Обработка запросов

Endpoint должен принимать только нужные HTTP-методы.

Для формы обычно нужен `POST`.

Если пришел другой метод, нужно вернуть корректный ответ:

```php
http_response_code(405);
header('Allow: POST');
```

Правила:

- проверять `Content-Type`, если endpoint ожидает JSON;
- ограничивать размер тела запроса;
- читать только ожидаемые поля;
- игнорировать или отклонять лишние поля;
- возвращать JSON для AJAX-запросов;
- не отдавать пользователю stack trace и внутренние ошибки.

## Валидация

Все данные от пользователя нужно валидировать на сервере.

Проверяй:

- обязательные поля;
- длину строк;
- email;
- телефон;
- согласие на обработку данных, если оно требуется;
- тип формы;
- скрытое honeypot-поле;
- размер и тип файлов;
- допустимые значения select, checkbox и radio.

Нельзя доверять HTML-валидации, React-валидации или Astro-форме. Клиентская валидация нужна для удобства, серверная — для безопасности.

## Ответы API

Для endpoint лучше возвращать единый формат JSON.

Пример:

```json
{
  "ok": true,
  "message": "Заявка отправлена"
}
```

Для ошибки:

```json
{
  "ok": false,
  "message": "Проверьте поля формы"
}
```

Правила:

- сообщения для пользователя должны быть понятными;
- технические детали не должны попадать в ответ;
- HTTP-статусы должны соответствовать результату;
- ошибки валидации можно возвращать по полям;
- успешный ответ не должен раскрывать внутренние ID, токены и служебные данные.

## Отправка писем

Для отправки писем лучше использовать SMTP или надежный сервис, а не голый `mail()`, если хостинг позволяет.

Правила:

- валидировать email отправителя;
- не использовать пользовательский email как технический `From`;
- пользовательский email можно ставить в `Reply-To`;
- экранировать значения в письме;
- не отправлять в письмо лишние персональные данные;
- не вставлять сырые пользовательские заголовки;
- обрабатывать ошибку отправки;
- логировать только безопасный минимум.

Если используется `mail()`, нужно особенно внимательно защищаться от header injection.

## Интеграции с CRM и webhook

Webhook и токены CRM должны быть только на сервере.

Правила:

- не отдавать webhook URL в клиентский JavaScript;
- проверять, какие поля уходят в CRM;
- не отправлять лишние персональные данные;
- задавать timeout для внешнего запроса;
- обрабатывать ошибки CRM;
- не считать заявку успешной, если критичная интеграция упала;
- логировать ошибку без токенов и полного payload с персональными данными.

Если заявка должна уйти и на почту, и в CRM, нужно явно решить, какой результат считается успешным.

## Защита от спама

Формы должны иметь базовую защиту от спама.

Возможные меры:

- honeypot-поле;
- минимальное время заполнения формы;
- rate limit по IP;
- ограничение частоты на endpoint;
- captcha, если спама много;
- серверная проверка обязательных полей;
- блокировка слишком длинных значений;
- журналирование подозрительных запросов без персональных данных.

Не нужно ставить тяжелую captcha на каждую форму заранее, если достаточно простого honeypot и rate limit.

## Загрузка файлов

Если форма принимает файлы, нужно проверять:

- размер файла;
- MIME type;
- расширение;
- реальный тип файла, если возможно;
- количество файлов;
- имя файла;
- директорию загрузки;
- права доступа.

Правила:

- не доверять имени файла от пользователя;
- генерировать безопасное имя на сервере;
- не разрешать загрузку PHP, HTML, JS и исполняемых файлов;
- хранить загруженные файлы вне публичной директории, если они не должны быть публичными;
- не выполнять загруженные файлы;
- очищать временные файлы.

## Работа с базой данных

Если PHP-код работает с базой данных, используй подготовленные запросы.

Правила:

- использовать PDO или безопасный database-layer проекта;
- не склеивать SQL из пользовательских данных;
- задавать кодировку соединения;
- проверять ошибки;
- не выводить SQL-ошибки пользователю;
- не хранить пароль базы в публичном файле;
- минимизировать права пользователя базы данных.

Для маленького лендинга база часто не нужна. Не добавляй ее без реальной задачи.

## Сессии и cookies

Если используются сессии или cookies, они должны быть настроены безопасно.

Правила:

- использовать `HttpOnly` для cookies, если они не нужны JavaScript;
- использовать `Secure` на HTTPS;
- задавать `SameSite`;
- не хранить в cookies секреты и персональные данные;
- регенерировать session ID после входа пользователя;
- задавать разумное время жизни.

Для простых форм сессии часто не нужны.

## CORS

CORS должен быть строгим.

Правила:

- не ставить `Access-Control-Allow-Origin: *`, если endpoint принимает персональные данные;
- разрешать только нужные домены;
- не разрешать лишние методы;
- не разрешать лишние headers;
- не включать credentials без необходимости.

Если форма отправляется с того же домена, CORS обычно не нужен.

## Логи

Логи нужны для диагностики, но они не должны становиться утечкой данных.

Можно логировать:

- время ошибки;
- тип ошибки;
- безопасный код ошибки;
- endpoint;
- технический статус внешней интеграции.

Нельзя логировать:

- пароли;
- токены;
- cookies;
- полные номера банковских карт;
- полные payload с персональными данными;
- содержимое загруженных файлов.

Логи должны быть закрыты от публичного доступа.

## Ошибки

В продакшене нельзя показывать PHP-ошибки пользователю.

Правила:

- `display_errors` должен быть выключен в продакшене;
- ошибки должны писаться в закрытый лог;
- пользователь должен видеть нейтральное сообщение;
- stack trace не должен попадать в HTML или JSON;
- разные внутренние ошибки не должны раскрывать структуру сервера.

В разработке ошибки можно показывать, если это не публичная среда.

## Права файлов

Права доступа должны быть минимально достаточными.

Правила:

- PHP-файлы не должны быть доступны на запись веб-серверу без причины;
- директории загрузок должны иметь отдельные ограничения;
- `.env`, конфиги и логи должны быть закрыты;
- не использовать `777` без крайней необходимости;
- резервные копии не должны лежать в публичной директории.

## Деплой на сервер

Перед деплоем нужно проверить:

- серверную версию PHP;
- наличие нужных расширений;
- корректность путей;
- права файлов и директорий;
- переменные окружения;
- работу SMTP или внешних API;
- доступность endpoint;
- что `.env`, логи и конфиги не открываются по URL;
- что тестовые файлы не попали в продакшен.

Если проект собирается Astro, PHP-часть должна быть развернута так, чтобы статический frontend обращался к правильному endpoint.

## Проверки

После изменения PHP-кода нужно запустить доступные проверки:

```bash
php -l path/to/file.php
composer test
composer run lint
```

Если PHP локально недоступен, нужно написать об этом в ответе.

Для обработчиков форм дополнительно проверить:

- успешную отправку;
- ошибку валидации;
- пустые поля;
- неверный email;
- повторную отправку;
- спам-поле honeypot;
- ошибку SMTP или CRM;
- ответ JSON;
- отсутствие секретов в ответе.

## Финальная проверка PHP-задачи

Перед завершением задачи Codex должен убедиться:

- PHP нужен для этой задачи;
- endpoint принимает только нужные методы;
- данные валидируются на сервере;
- секреты не попали в клиентский код и публичные файлы;
- письма и webhook защищены от инъекций и утечек;
- ошибки не раскрывают внутренние детали;
- логи не содержат чувствительные данные;
- публичная директория не содержит `.env`, логи и резервные копии;
- доступные проверки выполнены или причина их отсутствия указана в ответе.
