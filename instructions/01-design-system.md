# 01. Дизайн-система проекта

Этот файл описывает, где хранится дизайн-система проекта и как Codex должен использовать ее при работе над интерфейсом.

## Где хранить дизайн-систему

Дизайн-система конкретного проекта хранится в файле:

```text
docs/design-system.md
```

Если проект начинается с нуля, создай этот файл до активной верстки. Если дизайн-система уже есть в Figma, PDF, Notion, брендбуке или другом источнике, перенеси в `docs/design-system.md` только рабочие правила, которые нужны для разработки.

Файл `docs/design-system.md` должен быть главным источником правды для визуальной части проекта.

## Как Codex должен работать с дизайн-системой

Перед любой задачей, которая меняет интерфейс, Codex должен прочитать `docs/design-system.md` и свериться с ним.

Если дизайн-система заполнена, нельзя произвольно менять:

- цвета;
- шрифты;
- размеры контейнеров;
- радиусы;
- характер кнопок;
- стиль карточек;
- визуальный тон сайта;
- правила адаптива.

Если нужного правила нет, Codex выбирает решение, которое ближе всего к существующему стилю проекта, и не добавляет новый визуальный язык без необходимости.

## Правила для React и Astro

В React и Astro дизайн-система должна превращаться в повторно используемые сущности:

- CSS-переменные для цветов, размеров, радиусов, теней и z-index;
- общие классы или utility-слой для сетки и отступов;
- переиспользуемые компоненты для кнопок, полей, карточек и секций;
- единые состояния `hover`, `focus`, `active`, `disabled`, `loading`;
- единая логика адаптива.

Не нужно дублировать стили вручную в каждом компоненте, если их можно вынести в общий слой.

## Правила для CSS-переменных

Основные токены хранятся в глобальном CSS-файле проекта, например:

```text
src/styles/tokens.css
```

Минимальный набор токенов:

```css
:root {
  --color-bg: #ffffff;
  --color-text: #111111;
  --color-primary: #0057ff;
  --color-border: #e5e7eb;

  --font-base: system-ui, sans-serif;

  --container-width: 1200px;
  --space-section: 96px;

  --radius-sm: 4px;
  --radius-md: 8px;
}
```

Значения выше являются примером. В реальном проекте их нужно заменить на значения из `docs/design-system.md`.

## Если дизайн-системы еще нет

Если клиент или проект не дал дизайн-систему, Codex должен:

1. Создать базовую дизайн-систему в `docs/design-system.md`.
2. Выбрать спокойные, практичные визуальные правила под тип сайта.
3. Не использовать случайные декоративные решения.
4. Поддерживать единый стиль во всех секциях.
5. Позже обновить дизайн-систему, если появятся брендбук, макеты или новые требования.

## Проверка перед завершением задачи

Перед завершением UI-задачи проверь:

- интерфейс соответствует `docs/design-system.md`;
- цвета и типографика не разъехались между секциями;
- элементы имеют состояния `hover`, `focus` и `disabled`, где это нужно;
- на mobile нет наложений текста и элементов;
- изображения не ломают сетку;
- новые стили не конфликтуют с существующими токенами.
