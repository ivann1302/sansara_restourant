# 05. SEO и гео-ориентация сайта

Этот файл описывает, как Codex должен работать с SEO, локальной гео-ориентацией и структурой сайта для лендингов и корпоративных сайтов.

## Главный принцип

SEO должно помогать пользователю и поисковым системам понять страницу.

Codex не должен писать текст, структуру или метаданные только ради набора ключевых слов. Важнее:

- понятная структура страницы;
- полезный контент;
- соответствие интенту пользователя;
- быстрый сайт;
- доступность важного текста в HTML;
- корректные URL;
- нормальная внутренняя перелинковка;
- честная информация о компании, услугах, ценах, адресах и контактах.

SEO не должно ломать дизайн, UX, скорость и доверие к сайту.

## Перед работой

Перед SEO-правками нужно понять:

- для какой компании или проекта делается сайт;
- какие услуги или товары продвигаются;
- какие города, регионы или страны важны;
- есть ли физический адрес;
- работает ли компания удаленно, локально или по всей стране;
- какие страницы уже существуют;
- есть ли семантика, структура или ТЗ от SEO-специалиста;
- какие ограничения есть в дизайн-системе.

Если данных о регионе или услуге нет, не нужно выдумывать адреса, города, отзывы, лицензии, цены и факты.

## Базовая SEO-структура страницы

Каждая важная страница должна иметь:

- один понятный `h1`;
- логичную структуру `h2` и `h3`;
- уникальный `title`;
- уникальный `description`;
- человекочитаемый URL;
- основной текст, доступный в HTML;
- внутренние ссылки на связанные страницы;
- понятные CTA;
- контактную информацию, если она важна для интента;
- изображения с корректными `alt`;
- canonical URL, если есть риск дублей.

Нельзя делать несколько одинаковых страниц, которые отличаются только названием города, если контент не адаптирован под этот город.

## Title

`title` должен кратко описывать страницу и отличаться от других страниц.

Хорошо:

```text
Разработка корпоративных сайтов в Казани | Компания
```

Плохо:

```text
Главная
Лучшие услуги недорого быстро купить заказать
```

Правила:

- основной смысл страницы ставить ближе к началу;
- не набивать ключевыми словами;
- добавлять город, если страница действительно региональная;
- добавлять бренд, если это полезно;
- не делать одинаковые title на разных страницах.

## Description

`description` должен помогать пользователю понять, что он найдет на странице.

Правила:

- писать человеческим языком;
- кратко раскрывать услугу, выгоду или содержание страницы;
- добавлять город или регион, если это важно;
- не повторять title дословно;
- не обещать то, чего нет на странице;
- не использовать набор ключей через запятую.

## Заголовки

На странице должен быть один главный `h1`.

`h1` описывает основной смысл страницы:

```text
Разработка сайтов для строительных компаний
```

`h2` и `h3` должны помогать структуре:

- услуги;
- преимущества;
- этапы работы;
- цены;
- кейсы;
- FAQ;
- контакты;
- регионы работы.

Не нужно использовать заголовки как декоративный текст, если они не отражают структуру страницы.

## URL

URL должны быть короткими, стабильными и понятными.

Хорошо:

```text
/services/site-development/
/services/seo/
/cases/
/contacts/
/kazan/
/kazan/site-development/
```

Плохо:

```text
/page-1/
/new-final-seo/
/uslugi123/
/site-development-kazan-kupit-nedorogo-luchshee/
```

Если URL меняется у уже опубликованной страницы, нужно предусмотреть редирект со старого адреса.

## Важный контент должен быть в HTML

Для SEO важный контент не должен зависеть только от клиентского JavaScript.

В HTML должны быть доступны:

- заголовки;
- описания услуг;
- преимущества;
- цены или принципы расчета;
- FAQ;
- контакты;
- адреса;
- регионы работы;
- ссылки на важные страницы;
- хлебные крошки;
- тексты карточек услуг и кейсов.

Для Astro это означает: основной контент лучше отдавать через `.astro`, Markdown, Content Collections или серверный рендер, а React использовать только для интерактивности.

## Гео-ориентация

Гео-ориентация нужна, если бизнес работает в конкретном городе, регионе или стране.

На сайте должны быть явно указаны:

- город или регион работы;
- адрес, если есть офис или точка обслуживания;
- телефон;
- режим работы;
- способы связи;
- зона выезда или доставки;
- локальные особенности услуги, если они есть.

Город должен появляться там, где это естественно:

- в `title`;
- в `description`;
- в `h1` или первом экране;
- в контактах;
- в тексте услуги;
- в FAQ;
- в schema-разметке;
- в хлебных крошках;
- во внутренних ссылках.

Не нужно вставлять город в каждое предложение.

## Региональные страницы

Региональные страницы можно создавать, если для каждого региона есть реальная польза и отличия.

Хорошие отличия:

- локальный адрес или точка обслуживания;
- разные цены или условия;
- разные сроки;
- реальные кейсы в регионе;
- локальные отзывы;
- особенности доставки, выезда или обслуживания;
- отдельные контакты;
- отдельные фотографии или примеры работ.

Плохой подход:

- один и тот же текст с заменой города;
- десятки страниц без реального локального контента;
- выдуманные адреса;
- выдуманные отзывы;
- несуществующие офисы;
- страницы городов, где компания не работает.

Если контент для региональных страниц пока одинаковый, лучше сделать одну страницу с блоком регионов работы.

## Структура для регионального сайта

Возможные структуры:

```text
/contacts/
/services/
/services/site-development/
/regions/
/regions/kazan/
/regions/samara/
```

Или, если регион является главным входом:

```text
/kazan/
/kazan/site-development/
/samara/
/samara/site-development/
```

Выбор структуры зависит от масштаба проекта.

Для нескольких городов достаточно простой структуры. Для крупного мультирегионального сайта нужно заранее продумать шаблоны, sitemap, хлебные крошки и перелинковку.

## Локальная релевантность

Региональная страница должна отвечать на локальные вопросы пользователя.

Можно добавить:

- где компания оказывает услугу;
- сколько занимает работа в этом регионе;
- доступен ли выезд;
- есть ли доставка;
- какие районы обслуживаются;
- какие документы нужны;
- какие местные ограничения есть;
- локальные кейсы;
- карту или схему проезда;
- фотографии офиса, производства или объектов.

Не добавляй локальные детали, если они не подтверждены.

## Контакты

Контактная информация должна быть консистентной.

Проверь:

- название компании;
- адрес;
- телефон;
- email;
- мессенджеры;
- режим работы;
- реквизиты, если они нужны;
- ссылки на карты и профили компании.

Одинаковые данные должны совпадать в шапке, футере, странице контактов, schema-разметке и локальных блоках.

## Schema.org

Микроразметка должна помогать поисковым системам понять страницу.

Часто полезны:

- `Organization`;
- `LocalBusiness`;
- `WebSite`;
- `WebPage`;
- `BreadcrumbList`;
- `Service`;
- `FAQPage`;
- `Article`;
- `Product`, если есть товары;
- `Review`, если есть реальные отзывы и они разрешены правилами площадки.

Правила:

- schema должна соответствовать видимому контенту страницы;
- нельзя размечать то, чего нет на странице;
- нельзя выдумывать рейтинг, отзывы, адреса и цены;
- JSON-LD лучше держать рядом с layout или страницей;
- данные в schema должны совпадать с контактами на сайте.

## Open Graph

Для важных страниц нужно задавать Open Graph-данные:

- `og:title`;
- `og:description`;
- `og:type`;
- `og:url`;
- `og:image`;
- `og:site_name`.

Изображение для шаринга должно быть реальным, понятным и не ломаться при публикации в мессенджерах и соцсетях.

## Sitemap

Для продакшен-сайта нужен `sitemap.xml`.

В sitemap должны попадать:

- главная;
- страницы услуг;
- страницы регионов;
- кейсы;
- статьи;
- контакты;
- другие важные публичные страницы.

В sitemap не должны попадать:

- тестовые страницы;
- черновики;
- страницы благодарности после формы;
- служебные endpoint;
- страницы с noindex;
- дубли.

Для Astro обычно стоит использовать `@astrojs/sitemap`, если проекту это подходит.

## Robots и индексация

Для продакшена нужен корректный `robots.txt`.

Правила:

- тестовые окружения должны быть закрыты от индексации;
- продакшен не должен случайно иметь `Disallow: /`;
- служебные разделы нужно закрывать, если они существуют;
- sitemap должен быть указан в `robots.txt`;
- страницы с приватными или техническими данными не должны быть публичными.

Не полагайся только на `robots.txt` для защиты секретных данных. Секретные данные не должны попадать в публичную директорию вообще.

## Canonical и дубли

Canonical нужен, если одна и та же или очень похожая страница доступна по разным URL.

Проверь дубли:

- со слэшем и без слэша;
- с параметрами;
- http и https;
- www и без www;
- страницы сортировок и фильтров;
- одинаковые региональные страницы.

Canonical не должен указывать на нерелевантную страницу.

## Hreflang

`hreflang` нужен, если сайт имеет версии для разных языков или стран.

Правила:

- использовать только для реальных альтернативных страниц;
- ссылки должны быть взаимными;
- язык и регион должны быть указаны корректно;
- не использовать `hreflang` для обычных городских страниц на одном языке, если это не языковые или страновые версии.

## Изображения и alt

`alt` должен описывать смысл изображения.

Правила:

- для значимых изображений писать конкретный `alt`;
- для декоративных изображений использовать пустой `alt`;
- не набивать `alt` ключевыми словами;
- не писать `alt="картинка"` или `alt="фото"`;
- для локальных фото можно указать место, если оно реально важно.

Пример:

```html
<img src="/images/office-kazan.jpg" alt="Офис компании в Казани" />
```

## Внутренняя перелинковка

Важные страницы должны быть связаны между собой.

Полезные связи:

- главная → ключевые услуги;
- услуги → кейсы;
- услуги → FAQ;
- услуги → контакты;
- регион → услуги в регионе;
- кейс → услуга;
- статья → услуга;
- footer → контакты, услуги, регионы.

Не нужно добавлять блок ссылок без смысла для пользователя.

## FAQ

FAQ полезен, если вопросы реальные.

Хорошие вопросы:

- сколько стоит услуга;
- какие сроки;
- как начать работу;
- какие документы нужны;
- работаете ли в конкретном городе;
- есть ли выезд;
- какие гарантии;
- что входит в услугу.

Не нужно делать FAQ из искусственных SEO-вопросов, если они не помогают пользователю.

Если используется `FAQPage` schema, вопросы и ответы должны быть видимы на странице.

## Контент

Контент должен быть конкретным.

Хорошо:

- описывать реальные услуги;
- объяснять процесс;
- показывать ограничения;
- давать понятные CTA;
- использовать реальные преимущества;
- добавлять кейсы, цифры и факты, если они подтверждены.

Плохо:

- писать общие фразы без смысла;
- повторять ключевые слова;
- обещать гарантированный результат в поиске;
- выдумывать опыт, клиентов, рейтинги и отзывы;
- копировать одинаковые блоки между страницами.

## Техническая SEO-проверка

Перед завершением SEO-задачи проверь:

- у страницы есть один `h1`;
- `title` и `description` уникальны;
- важный контент есть в HTML;
- URL понятный;
- canonical корректный, если он нужен;
- изображения имеют корректные `alt`;
- внутренние ссылки работают;
- нет лишнего `noindex`;
- страница не закрыта в `robots.txt`;
- sitemap содержит нужные страницы;
- schema соответствует видимому контенту;
- mobile-верстка не скрывает важный контент;
- нет критичных ошибок сборки.

## GEO для AI-поиска

Если под GEO имеется в виду Generative Engine Optimization, а не география, Codex должен дополнительно следить, чтобы контент был понятен AI-поиску и ответным системам.

Правила:

- ясно описывать, кто оказывает услугу;
- явно указывать город, страну и зону работы;
- давать прямые ответы на частые вопросы;
- использовать структурированные блоки: услуги, цены, этапы, FAQ, контакты;
- не прятать важные факты в изображениях;
- поддерживать актуальные данные о компании;
- добавлять schema-разметку, где она уместна;
- избегать пустых маркетинговых фраз без проверяемого смысла.

Этот раздел не заменяет обычное SEO. Он только делает контент более понятным для систем, которые формируют ответы на основе страниц.

## Финальная проверка SEO/GEO-задачи

Перед завершением задачи Codex должен убедиться:

- страница отвечает на интент пользователя;
- регион указан честно и уместно;
- нет выдуманных адресов, отзывов и фактов;
- важный контент доступен без клиентского JavaScript;
- метаданные уникальны;
- локальные данные совпадают в тексте, контактах и schema;
- региональные страницы не являются пустыми дублями;
- sitemap, robots и canonical не мешают индексации;
- изменения не ухудшили скорость, доступность и мобильную верстку.
