# AGENTS.md

Этот файл является главным входом для Codex в проекте.

Codex должен сначала прочитать этот файл, определить тип задачи и затем открыть только те подробные инструкции из папки `instructions/`, которые относятся к задаче.

## Базовый контекст проекта

Этот стартер предназначен для лендингов и корпоративных сайтов.

Основной стек:

- Astro для статических страниц, SEO и быстрой загрузки;
- React для интерактивных островов, форм, калькуляторов и сложных виджетов;
- PHP на сервере только там, где нужна серверная логика: формы, почта, CRM, webhook, загрузка файлов;
- CSS и компоненты должны следовать дизайн-системе проекта.

## Главные правила

- Сначала понять структуру проекта, потом менять код.
- Делать минимальные изменения под задачу.
- Соблюдать KISS и YAGNI.
- Использовать SOLID только там, где он реально упрощает поддержку сложной логики.
- Не добавлять зависимости без необходимости.
- Не превращать Astro-сайт в SPA без причины.
- Не использовать React для простой статической верстки.
- Не хранить секреты в клиентском коде.
- Не выводить пользователю технические ошибки.
- Не логировать токены, пароли, cookies и полные данные лидов.
- После изменений запускать доступные проверки или объяснять, почему они недоступны.

## Как выбирать инструкции

Всегда учитывай общие правила:

- `instructions/03-general-programming.md`

Если задача касается интерфейса, верстки, компонентов, стилей, адаптива или визуального качества, прочитай:

- `instructions/01-design-system.md`

Если задача касается React, React-компонентов, хуков, форм на React или интерактивных виджетов, прочитай:

- `instructions/02-react-rules.md`

Если задача касается Astro, страниц, layouts, маршрутов, content collections, гидрации или сборки сайта, прочитай:

- `instructions/04-astro-rules.md`

Если задача касается SEO, GEO, региональных страниц, метаданных, sitemap, robots, schema.org, Open Graph или структуры контента, прочитай:

- `instructions/05-seo-geo.md`

Если задача касается PHP, серверных endpoint, отправки писем, CRM, webhook, загрузки файлов, Composer или серверного деплоя, прочитай:

- `instructions/06-php-server.md`

Если задача касается форм, лидов, секретов, токенов, `.env`, логов, CORS, HTTPS, публичной директории, сторонних скриптов или защиты от спама, прочитай:

- `instructions/07-security.md`

## Приоритет правил

Если инструкции конфликтуют:

1. Безопасность важнее удобства реализации.
2. Данные пользователя и лиды важнее логов и аналитики.
3. Дизайн-система важнее случайных визуальных решений.
4. SEO-контент должен быть доступен в HTML, если это возможно.
5. Более конкретная инструкция важнее общей.
6. Явная задача пользователя важнее стандартных рекомендаций, если она не нарушает безопасность.

## Дизайн-система проекта

Дизайн-система конкретного проекта должна находиться в:

```text
docs/design-system.md
```

Если файла нет, а задача касается интерфейса, Codex должен создать или предложить базовую структуру дизайн-системы по правилам из `instructions/01-design-system.md`.

## Защита лидов

Любая форма, которая собирает лиды, должна отправлять данные на серверный endpoint.

Нельзя раскрывать в браузере:

- CRM webhook;
- SMTP-доступы;
- API-ключи;
- приватные токены;
- внутренние ответы CRM;
- полные технические ошибки.

Минимальная защита лидов:

- server-side validation;
- honeypot или другой простой антиспам, если форма публичная;
- ограничение лишних полей;
- отправка только в согласованные каналы;
- аккуратная обработка ошибок;
- безопасные логи без полного текста заявки и секретов.

## Проверки перед завершением

Перед финальным ответом Codex должен проверить:

- задача действительно решена;
- изменения соответствуют нужным инструкциям;
- нет случайного усложнения;
- секреты не попали в клиентский код;
- формы и лиды не раскрывают лишние данные;
- важный SEO-контент не спрятан за клиентским JavaScript;
- доступные проверки выполнены.

Если проверки не запускались, нужно кратко объяснить почему.

## Формат финального ответа

Финальный ответ должен быть коротким и полезным:

- что изменено;
- где изменено;
- какие проверки запускались;
- что осталось учесть, если есть ограничения.

Не нужно писать длинный отчет для маленьких задач.
